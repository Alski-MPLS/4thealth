4THealth Developer Guide

This document is the technical reference for developers contributing to, deploying, or extending 4THealth. It covers how the application is structured, how each authentication mode works, all API endpoints, and step-by-step procedures for both local development and production deployment.

Read-only by design. 4THealth never writes to FortiManager or any FortiGate device. Every API call uses get or monitor methods only.

What is 4THealth?

4THealth is a Flask web dashboard that surfaces live health data for a Fortinet infrastructure stack. All data comes through FortiManager's JSON-RPC API — the app has no direct connection to individual FortiGate firewalls.

It provides six main views:

ViewWhat it shows
DashboardInfrastructure health cards for FortiManager (×2), FortiAnalyzer (×2), and FortiCollector (×2) — hostname, version, serial, HA role, CPU %, memory %, disk
FirewallsPer-ADOM device list with health status, paginated table, full device detail modal (interfaces, routes, BGP, OSPF, IPsec)
Device VersionsVersion distribution chart per ADOM with clickable filtering and CSV/JSON export
Rule ReviewPolicy-package audit — 7 read-only checks with paginated findings and CSV/JSON/PDF export
AdminGroup and tab-permission management; in-memory application log viewer
HelpIn-app contextual help panel accessible from the nav bar

Blueprint Registration

app/__init__.py creates the Flask app via the create_app() factory. Blueprints are loaded from a list in _BLUEPRINT_MODULES — adding a new module is one line there:

_BLUEPRINT_MODULES = [
    "app.routes.auth_routes",
    "app.routes.dashboard_routes",
    "app.routes.api_routes",
    "app.routes.admin_routes",
    "app.routes.hygiene_routes",
    "app.routes.my_new_module",   # ← add here only
]

The factory also injects three Jinja2 template globals after every blueprint is loaded:

GlobalTypeDescription
current_rolestr"admin" or "viewer" from session
allowed_tabsset[str]Tab keys the current user can access
nav_registrydictFull registry snapshot used by base.html to render the nav bar — populated automatically by registry.register() calls in each blueprint

Data Flow

Browser │ │ HTTPS (Nginx → Gunicorn in production; wsgi.py in dev) ▼ Flask Route ──► FMGClient.get_*() │ │ JSON-RPC POST /jsonrpc │ (Bearer token, session-based) ▼ FortiManager │ │ /sys/proxy/json (REST proxy to managed device) ▼ FortiGate device (read-only REST monitor endpoints) │ └──► Response unwrapped through two envelope layers and returned to the Flask route as a Python dict

FortiManager Client Design

app/fmg_client.py implements a context manager:

with FMGClient(host, username, password, verify_ssl=False) as client:
    devices = client.get_devices("root")
    health  = client.get_device_health("root", "fw-prod-01")

On __enter__ the client calls login() to obtain a session token via FortiManager's /jsonrpc endpoint. On __exit__ it calls logout(). All device telemetry flows through the FortiManager proxy endpoint — the app never directly connects to individual firewalls.

The proxy envelope is double-wrapped (FMG adds an outer envelope; the device adds an inner one). The client's _proxy() method unwraps both layers transparently.

Health Status Model

Three-tier status is applied after metrics are collected:

StatusConditionCSS class
🟢 GreenCPU < CPU_WARN and MEM < MEM_WARNstatus-green
🟡 YellowEither metric crossed WARN thresholdstatus-yellow
🔴 RedEither metric crossed CRIT threshold, or device unreachablestatus-red

Default thresholds: CPU 70% warn / 90% crit; MEM 75% warn / 90% crit. Override in .env.

Quick Start — Local Development

Development uses local bcrypt accounts stored in users.json. No Active Directory or network connectivity to Fortinet devices is required to start the server, but you will need FortiManager credentials to see live data.

Prerequisites

  • Python 3.11+
  • uv — brew install uv on macOS
  • A .env file (see Configuration Reference below)

Step-by-Step

# 1. Install dependencies into an isolated .venv
uv sync

# 2. Copy and edit the environment file
cp .env.example .env
# Minimum required: set SECRET_KEY, FMG_PASSWORD

# 3. Generate a strong SECRET_KEY
uv run python manage_users.py secret
# Copy the output into .env as: SECRET_KEY=<value>

# 4. Create the first admin account
uv run python manage_users.py add admin --role admin

# 5. Start the dev server
uv run python wsgi.py
# Listening on http://localhost:5000

# Optional: enable HTTPS locally
mkdir -p certs
openssl req -x509 -newkey rsa:4096 -nodes \
  -keyout certs/key.pem -out certs/cert.pem \
  -days 3650 -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"
uv run python wsgi.py
# Listening on https://localhost:5443

Never prefix commands with python directly. Always use uv run python … so the correct virtual environment is used. Do not use pip install — use uv add <package> to keep uv.lock in sync.

Local Authentication (Development Mode)

In development (and as an emergency fallback in production), authentication is handled by app/auth.py using bcrypt password hashing.

How It Works

  1. User submits username + password via POST /login.
  2. auth_routes.py calls authenticate(username, password).
  3. auth.py loads users.json, finds the user record, and calls bcrypt.checkpw().
  4. On success, the session is populated: session["user"], session["role"], session["allowed_tabs"].
  5. The user is redirected to the ?next= parameter (safe open-redirect validation applied) or /.

Session Contents

KeyTypeDescription
session["user"]strUsername
session["role"]str"admin" or "viewer"
session["allowed_tabs"]list[str]Tab keys the user may access (empty for admin — admin gets all)

Sessions are signed with SECRET_KEY using Flask's itsdangerous library. Clients cannot forge or tamper with the session cookie.

Groups and Tab Permissions

Groups are managed through the Admin → Groups & Permissions UI. Group definitions are stored in groups.json (gitignored — copy from groups.example.json to get started).

The logic in app/groups.py:

  • admin role → always gets all tabs regardless of group membership.
  • viewer role → gets the union of allowed_tabs across every group they belong to.
  • A viewer with no group memberships sees no content tabs.

KNOWN_TABS Registry

All navigable tabs are registered in app/groups.py:

KNOWN_TABS: dict[str, str] = {
    "dashboard":    "Dashboard",
    "firewalls":    "Firewalls",
    "versions":     "Device Versions",
    "rule_hygiene": "Rule Review",
}

Adding a tab here makes it appear immediately in the Admin group-editor checklist. Protect the corresponding route with @tab_required("your_tab_key").

Application Logging

app/app_logger.py provides an in-memory ring-buffer logger. It is viewable in real time through Admin → Application Logs without tailing a file.

LevelUse when
ERRORUnhandled exceptions, authentication failures
WARNFailed login attempts, unexpected API responses
INFO (default)Login/logout events, group changes
DEBUGAdmin page access, API round-trips
TRACEDetailed per-request data for deep troubleshooting
  • Max 2 000 entries — oldest are dropped when full. Buffer is cleared on process restart.
  • Active capture level controls what is written — entries below the level are discarded immediately, not just hidden.
  • Level can be changed at runtime from the Admin UI (no restart needed).
  • Thread-safe with threading.Lock; safe under multi-worker gunicorn (each worker has its own buffer).

Production Overview

In production 4THealth runs behind Nginx as a reverse proxy, with gunicorn serving the WSGI application. See production.md for the full step-by-step guide. This section covers the authentication and RBAC differences from development.

Internet / Internal Network │ HTTPS :443 ▼ Nginx (TLS termination, security headers, rate limiting) │ HTTP :8000 (127.0.0.1 only) ▼ Gunicorn (4 workers, timeout 120s) │ ▼ Flask app (wsgi:app) │ ├──── Local bcrypt auth (users.json fallback — always available) │ └──── Active Directory LDAP auth (when AD_ENABLED=true) │ │ ldaps://:636 ▼ Domain Controller (bind → user search → re-bind → memberOf)

Active Directory Authentication (Production)

When AD_ENABLED=true, the login flow changes:

  1. User submits credentials via POST /login.
  2. auth_routes.py calls the AD-aware authenticate() function.
  3. The function binds to the domain controller using the service account (AD_BIND_USER / AD_BIND_PASSWORD).
  4. It searches for the user by sAMAccountName in AD_USER_SEARCH.
  5. It re-binds using the user's credentials to validate the password.
  6. It reads the user's memberOf attribute and checks against AD_GROUP_ADMIN / AD_GROUP_VIEWER to determine role.
  7. On success, session is populated identically to local auth.

Always keep a local admin fallback. If the AD server is unreachable, local users.json accounts are the only way in. Emergency procedure: set AD_ENABLED=false in .env, restart the service, log in with the local admin account, restore after AD recovery.

AD Group → Tab Permission Mapping

When AD is active, group membership for tab permissions is resolved at login from the user's memberOf attribute (CN values), not from the members arrays in groups.json. The tab permission sets in groups.json remain authoritative — only the membership source changes.

Group names in groups.json must match the AD group CN values (or role strings) for the mapping to work. See production.md → Migrating Local Groups to Active Directory for the full migration checklist.

RBAC & Tab Permissions

RBAC is fully implemented — no code changes are needed at deployment time. This section explains the enforcement model.

Role Summary

RoleDashboardFirewallsVersionsRule ReviewAdmin pageRaw/Debug endpoints
admin AlwaysAlwaysAlwaysAlways ✅✅
viewer Per groupPer groupPer groupPer group ❌ 403❌ 403

Enforcement Layers

  1. Session cookie: Signed with SECRET_KEY via itsdangerous — clients cannot tamper with role or tab list.
  2. Route decorators: @login_required (all routes), @tab_required("key") (tab pages), @_admin_required (admin and raw endpoints).
  3. Template globals: current_role and allowed_tabs hide nav links cosmetically — but server-side enforcement is the real gate.

Security note: Nav link hiding is cosmetic only. A viewer who manually types /admin receives HTTP 403 regardless of what the nav bar shows.

Production Deployment Checklist

Full instructions are in production.md. This is the short verification checklist after deployment:

  • ☐ SECRET_KEY is a 64-char random hex string (not the example value)
  • ☐ .env permissions are 640, owner 4thealth:4thealth
  • ☐ users.json permissions are 640, owner 4thealth:4thealth
  • ☐ COOKIE_SECURE=true is set in .env
  • ☐ Nginx HSTS header is present (Strict-Transport-Security)
  • ☐ TLS 1.0/1.1 are disabled in Nginx (ssl_protocols TLSv1.2 TLSv1.3)
  • ☐ Port 8000 is not exposed externally (--bind 127.0.0.1:8000)
  • ☐ fail2ban is running and watching /login
  • ☐ Log rotation is configured under /etc/logrotate.d/4thealth
  • ☐ At least one local admin account exists in users.json as an AD fallback
  • ☐ Smoke test: curl -sk https://4thealth.yourdomain.com/login | grep -i 4thealth
  • ☐ RBAC test: log in as a viewer, confirm /admin returns 403

Module Design — How Extensibility Works

Three infrastructure modules make adding a new tab a self-contained operation that touches no existing files except one line in __init__.py:

ModuleWhat it provides
app/registry.py Tab registry. Each blueprint calls registry.register(key, name, endpoint) at import time. The app factory injects the resulting nav_registry dict into every template, so base.html renders the nav bar from data — no hard-coded tab blocks needed.
app/decorators.py Shared route decorators. login_required, tab_required("key"), and admin_required — import from here instead of redefining them in every blueprint. Each handles both browser requests (redirect to /login, HTTP 403 page) and API requests (JSON 401/403 responses) automatically.
app/fmg_helpers.py Shared FMG client factory. make_client() returns a configured FMGClient pointed at the primary FortiManager host. Import this instead of duplicating the host/auth config in every blueprint.

The admin UI tab-permission checklist is also fully automatic: on startup, the factory copies the registry into groups.KNOWN_TABS, which drives the GET /admin/api/tabs endpoint that populates the Admin group-editor checkboxes. No manual edit to groups.py needed.

Summary: to add a tab you create one new file (my_module.py) and add one line to __init__._BLUEPRINT_MODULES. The nav bar, Admin permission editor, and RBAC enforcement all update automatically.

Adding a New Tab / Module — Step by Step

Step 1 — Create the blueprint file

Create app/routes/my_module.py. The file has three responsibilities: register the tab, define the page route, and define API data routes.

"""My Module — brief description of what this tab does."""

from flask import Blueprint, render_template, session, jsonify
from app.decorators import tab_required
from app.fmg_helpers import make_client
from app.fmg_client import FMGError
from app import registry

bp = Blueprint("my_module", __name__)

# ── Register this tab ──────────────────────────────────────────────────────────
# key:      used in allowed_tabs, groups.json, and @tab_required()
# name:     shown in nav bar and Admin permission checklist
# endpoint: Flask endpoint string for url_for()
registry.register("my_module", "My Module", "my_module.my_page")


# ── Page route ─────────────────────────────────────────────────────────────────
@bp.route("/my-module")
@tab_required("my_module")
def my_page():
    return render_template("my_module.html", user=session["user"])


# ── Data API routes ────────────────────────────────────────────────────────────
@bp.route("/api/my-module/data")
@tab_required("my_module")
def my_data():
    try:
        with make_client() as client:
            raw = client.get_adoms()   # replace with real call
        return jsonify(raw)
    except FMGError as exc:
        return jsonify({"error": str(exc)}), 502
    except Exception as exc:
        return jsonify({"error": str(exc)}), 500

Step 4 — Create the JavaScript

Create app/static/js/my_module.js. Use an IIFE to avoid polluting global scope, mirror the patterns in versions.js or hygiene.js:

(function () {
  'use strict';

  // ── Populate ADOM dropdown ──────────────────────────────────────────────────
  async function loadAdoms() {
    const res = await fetch('/api/adoms');
    if (!res.ok) return;
    const adoms = await res.json();
    const sel = document.getElementById('adomSelect');
    adoms.forEach(a => sel.add(new Option(a, a)));
  }

  // ── Load data ───────────────────────────────────────────────────────────────
  async function load() {
    const adom = document.getElementById('adomSelect').value;
    if (!adom) return;
    const res = await fetch(`/api/my-module/data?adom=${encodeURIComponent(adom)}`);
    if (!res.ok) { showError('Failed to load data.'); return; }
    const data = await res.json();
    render(data);
  }

  function render(data) {
    document.getElementById('resultsContainer').innerHTML =
      `<pre>${JSON.stringify(data, null, 2)}</pre>`;
  }

  function showError(msg) {
    document.getElementById('resultsContainer').innerHTML =
      `<div class="alert alert-danger">${msg}</div>`;
  }

  document.getElementById('btnLoad').addEventListener('click', load);

  loadAdoms();
})();

Step 5 — Grant access via Admin

Start the server. Navigate to Admin → Groups & Permissions. The new tab "My Module" appears automatically in the tab checklist for every group. Check it for the groups that should have access, then save.

Admin users see the new tab immediately with no group changes needed — admins always have full access. Viewer users see it only after their group is updated.

Nav bar order

Tabs appear in the nav bar in the order their registry.register() calls execute. Blueprints are loaded in _BLUEPRINT_MODULES order, and within each blueprint, register() calls run top-to-bottom. Reorder entries in _BLUEPRINT_MODULES or move register() calls within a blueprint to change display order.

Optional: icon in nav label

registry.register("my_module", "My Module", "my_module.my_page", icon="📊")

The icon is prepended to the display name in the nav bar. Leave blank (default) for no icon.

Admin-only modules

If the new module should be admin-only (never shown to viewers), use @admin_required instead of @tab_required() and do not call registry.register() — the link will never appear in the nav bar for viewers. For an admin-only API endpoint on an existing blueprint, just add @admin_required:

from app.decorators import admin_required

@bp.route("/api/my-module/debug")
@admin_required
def my_debug():
    ...

registry.get_registry()   # {key: {name, endpoint, icon}, ...}
registry.known_tabs()     # {key: name, ...}  — compatible with groups.KNOWN_TABS

app/app_logger.py

from app.app_logger import app_log

app_log("INFO",  "my_module", "Something happened", user=username)
app_log("WARN",  "my_module", "Unexpected response", status=r.status_code)
app_log("ERROR", "my_module", "FMG call failed",    error=str(exc))
app_log("DEBUG", "my_module", "Request detail",      adom=adom, pkg=pkg)

The first two arguments are level and component. Use your module's name as the component so admins can filter the log by it. Extra keyword arguments appear in the "extra" column of the log viewer.

Adding a Hygiene Check

All hygiene checks live in app/hygiene.py as standalone functions. Each receives the full policy list and returns a list of finding dicts:

# app/hygiene.py

def check_my_rule(policies: list[dict]) -> list[dict]:
    findings = []
    for p in policies:
        seq  = p.get("policyid", "?")
        name = p.get("name", "")
        if <your condition here>:
            findings.append({
                "policy_id":   seq,
                "policy_name": name,
                "seq":         seq,
                "check":       "my_rule",
                "detail":      "Human-readable explanation of the finding",
            })
    return findings

Then add it to the CHECKS dict at the bottom of hygiene.py (this is what populates the hygiene UI checklist):

CHECKS: dict[str, tuple[str, Callable]] = {
    ...
    "my_rule": ("My Rule Display Name", check_my_rule),
}

The new check appears automatically in the Hygiene UI checklist. No changes needed to hygiene_routes.py, hygiene.html, or hygiene.js.

Frontend Conventions

No Build Tools

The entire frontend is plain HTML, CSS, and vanilla JavaScript. No webpack, no TypeScript, no React, no npm. This is intentional — it keeps the project dependency surface minimal and deployable anywhere Python runs.

Themes

style.css uses CSS custom properties on :root for both light and dark themes. The data-theme="dark" attribute is toggled on <html> by an inline script in base.html. Your templates inherit this automatically — never hard-code colors; use the CSS variables.

JavaScript Patterns

  • Wrap all module JS in an IIFE: (function () { 'use strict'; ... })();
  • Fetch data from /api/* using fetch(). Check res.ok before parsing.
  • Render tables by building an HTML string and assigning to element.innerHTML. Always run user-controlled strings through an esc() helper (see admin.js for the pattern).
  • Client-side pagination: hold the full dataset in a module-level array; renderTable() slices it by current page. The hygiene and versions modules are the clearest examples.
  • Export CSV: build a Blob, create a temporary <a>, set href + download, click it. Export PDF: open a new window, write a printable HTML page, call window.print().
  • Cache-bust static assets by incrementing the ?v=N query string in the <script src> tag when you make changes users have already cached.
